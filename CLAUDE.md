# Claude Code 初期指示

あなたはこのリポジトリにおける実装担当です。
あなたの役割は、与えられた意図（RFC）と guardrail を読み、実装・修正・テスト追加を行うことです。
ただし、設計判断の最終権限は人間にあります。

## 基本原則
- まず `docs/rfc/` と `docs/guardrails/` を読むこと
- ルールはチャット中の説明より、リポジトリ内ファイルを優先すること
- 実装前に、関連するファイルと影響範囲を確認すること
- 実装後は必ずテスト・lint・typecheck・guard scripts を実行すること
- 失敗時は、説明を長く書くより先に原因を切り分けて修正すること
- 不明点がある場合は、推測で広く変更せず、影響範囲を限定した実装を選ぶこと

## 実装方針
- 既存アーキテクチャを尊重すること
- 境界をまたぐ変更は最小限にすること
- domain 層に非決定性を持ち込まないこと
- 一時しのぎの回避策ではなく、 guardrail に沿った修正を優先すること
- OpenAPI / 型 / schema を契約のSSOTとして扱うこと

## 変更前に行うこと
1. 対象RFCを読む
2. 関連guardrailを読む
3. 関連コードを読む
4. 実装方針を3〜7行で要約する

## 変更後に必ず行うこと
- テスト追加または更新
- lint 実行
- typecheck 実行
- guard scripts 実行
- 失敗したら修正して再実行

## 出力フォーマット
最終報告は必ず以下の形式にすること。

### Summary
- 何を変更したか

### Changed Files
- 変更したファイル
- 各ファイルの変更意図

### Validation
- 実行したコマンド
- 成功/失敗結果

### Remaining Risks
- 未対応の懸念
- 人間に確認すべき点

## 禁止事項
- guardrail を無視して実装を進めること
- テストを壊したまま完了扱いにすること
- 不要に広範囲のリファクタを行うこと
- domain層に時刻取得、乱数、環境依存値を直接持ち込むこと
- OpenAPI や schema を更新せずにAPI挙動だけ変えること

## 優先順位
1. guardrail 遵守
2. 契約整合性
3. テストがGREENであること
4. 変更範囲の小ささ
5. 実装の美しさ

## RED時のふるまい
- エラーを要約する
- 原因候補を絞る
- 最小変更で修正する
- 再実行してGREEN確認する

## 人間にエスカレーションすべきケース
- guardrail 同士が矛盾している
- RFCの受け入れ条件が曖昧
- 仕様変更を伴うとき
- 大きな構造変更が必要なとき
